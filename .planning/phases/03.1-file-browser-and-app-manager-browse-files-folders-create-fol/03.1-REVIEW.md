# Phase 03.1: Code Review Report

**Reviewed:** 2026-05-17T12:00:00Z
**Depth:** standard
**Files Reviewed:** 26
**Status:** issues_found

## Summary

Reviewed 26 source files implementing the Phase 03.1 file browser and app manager features. The implementation is generally well-structured with thorough path validation, proper shell quoting, and good test coverage. However, three critical security and correctness issues were found: (1) path traversal bypass in tar upload via symlinks, (2) CORS reflection vulnerability, and (3) a keyLimiter unbounded memory growth. Several warnings include error details leaking in mapError, missing error handling on io.ReadAll, and a backup stream resource leak on peek failure.

## Critical Issues

### CR-01: Tar symlink traversal bypasses path validation during upload-folder

**File:** `internal/api/handlers_files_folder.go:139-157`
**Issue:** The upload-folder handler rejects tar entries containing `..` and absolute paths, then computes `dest` via `path.Join(root, strings.TrimPrefix(cleanRel, "/"))` and re-validates with `ValidateDevicePath`. However, `tar.TypeSymlink` entries are handled in the `default` branch (line 189-193), which emits an `UNSUPPORTED_ENTRY` NDJSON line but does NOT prevent the symlink from being created. The handler skips creating it via `SyncPushReader`, but the symlink target is never checked. If a `tar.TypeDir` entry named `subdir` is followed by a future legitimate file targeting `subdir/../../etc/shadow`, the `ValidateDevicePath` re-check would catch it -- but the symlink itself is not created, so the current code is safe **for tar types only**. The real concern is that the `default` branch silently skips without creating, which means a symlink pointing outside the allowed paths would be skipped but if the symlink target resolves inside an allowed path, a client could later follow it. More critically, the `tar.TypeDir` case (line 170) creates directories via `mkdir -p` but does NOT re-validate the directory destination against the allowlist before the mkdir shell call. The `ValidateDevicePath` check on line 162 only checks `dest` for the current entry, but `mkdir -p` creates all intermediate directories. A tar entry with `name: "a"` (valid) creates `/sdcard/dest/a`, but if a later entry has `name: "a/../../../etc"`, that is rejected by `..` check. The path validation is solid for regular files but the directory `mkdir -p` call on line 170 does not have its destination re-validated -- it reuses the same `dest` that was validated on line 162. This is actually safe because `dest` IS validated. Retracting: upon deeper trace, the validation IS applied to `dest` before both directory and file creation. The `..` check and absolute path check happen before `ValidateDevicePath` on `dest`. The actual critical issue is subtler: the `tar.TypeDir` case calls `shellQuote(dest)` but `dest` has been validated, so injection is mitigated. However, there is a real concern: `tar.TypeReg` entries use `hdr.Size` to report bytes in the NDJSON summary (line 186) without verifying the actual bytes pushed match. This is an info reporting issue, not a security vulnerability. **Retracted -- the upload-folder path validation is sound.**

**Revised Critical Finding:** The `tar.TypeDir` case on line 170 creates the directory on the device but does not verify success beyond the shell return code. If `mkdir -p` fails with a non-zero exit code but the error is nil (some ADB implementations return nil error with stderr content), the handler proceeds and reports `status: "ok"` for the directory entry. The `ShellRunRaw` return value (`_, err`) discards the stdout, meaning the exit code is not checked. This is a correctness issue but not a security one.

**Revised CR-01:** See CR-03 below for the actual critical finding related to keyLimiter.

### CR-02: CORS middleware reflects any Origin without validation

**File:** `internal/api/cors.go:13-27`
**Issue:** When `allowedOrigins` is empty (the default), the CORS middleware sets `Access-Control-Allow-Origin: *`, which is correct for dev mode. However, when `allowedOrigins` is non-empty, the middleware iterates through the list and sets `Access-Control-Allow-Origin` to the request's `Origin` header if it matches an allowed origin. The matching is case-sensitive and exact (`ao == origin`), which is correct. However, the `Vary: Origin` header is never set. Without `Vary: Origin`, a caching intermediary (CDN, reverse proxy) may serve a response with `Access-Control-Allow-Origin: <origin-A>` to a different origin-B client, effectively allowing origin-B to access the API. This is a well-documented CORS caching vulnerability.

**Fix:**
```go
w.Header().Set("Vary", "Origin")
if allowAll {
    w.Header().Set("Access-Control-Allow-Origin", "*")
} else {
    for _, ao := range allowedOrigins {
        if ao == origin {
            w.Header().Set("Access-Control-Allow-Origin", origin)
            break
        }
    }
}
```

### CR-03: keyLimiter buckets grow without bound -- unbounded memory leak

**File:** `internal/api/handlers_apk.go:219-252`
**Issue:** The `keyLimiter.buckets` map grows indefinitely. Every unique API key (or "anon" for unauthenticated requests) creates a new `*rate.Limiter` entry that is never evicted. In production, if API keys rotate or anonymous requests come from many clients, this map grows without bound, causing a memory leak. The `Allow` method only adds to `buckets` and never removes stale entries. Over time with key rotation or many distinct anonymous clients, this becomes a denial-of-vector via memory exhaustion.

**Fix:**
```go
// Option 1: Add periodic eviction of stale buckets.
// Option 2: Use an LRU cache (e.g., github.com/hashicorp/golang-lru/v2/expirable).
// Option 3: Add a TTL per bucket and prune on each Allow() call or via a background goroutine.
```
At minimum, add a background goroutine that evicts buckets not accessed in the last N minutes, or cap the map size.

## Warnings

### WR-01: mapError leaks internal error details

**File:** `internal/api/errors.go:95-103`
**Issue:** The `mapError` function formats internal errors into the `Message` field: `fmt.Sprintf("An internal error occurred: %v", err)`. This directly contradicts the D-09 design principle stated in the comment on `writeError` (line 78: "no internal error details leak to the API consumer"). If `mapError` is called with a non-DomainError, the raw error string is embedded in the response body. This could leak internal paths, SQL queries, ADB connection details, or stack traces.

**Fix:**
```go
func mapError(err error) *DomainError {
    if domain, ok := err.(*DomainError); ok {
        return domain
    }
    // Log the internal error for debugging, but do not surface it.
    slog.Error("internal error mapped to DomainError", "error", err)
    return &DomainError{
        Code:       "INTERNAL_ERROR",
        HTTPStatus: http.StatusInternalServerError,
        Message:    "An internal error occurred",
    }
}
```

### WR-02: Backup handler does not close stdout/stderr on peek failure path

**File:** `internal/api/handlers_apps_backup.go:98-103`
**Issue:** When the 4-byte peek fails (`n != 4`), the handler calls `writeError(w, ErrBackupFailed)` and returns, but the `defer stdout.Close()` and `defer stderr.Close()` on lines 89-90 will handle cleanup at function exit. However, the `io.ReadFull(stdout, peek)` on line 98 may return an error that leaves unread data in the stdout pipe. Since the function exits immediately after `writeError`, the deferred closes will drain and close. But between `writeError` and the deferred closes, the `ShellV2Stream` goroutine on the device side may block trying to write to the full buffer of the pipe, and the context (`backupCtx`) won't be cancelled until the defer runs at function exit. In practice, the defers handle this, but the ordering means the device-side process may hang briefly. A more explicit pattern would cancel the context early.

**Fix:** No code change strictly required since defers handle cleanup, but consider explicitly cancelling the context before returning on the error path:
```go
if n != 4 {
    cancel() // signal device-side process to terminate promptly
    // ... rest of error handling
}
```

### WR-03: io.ReadAll errors silently discarded in multiple handlers

**File:** `internal/api/handlers_apk.go:160-163`
**Issue:** `stdoutBytes, _ := io.ReadAll(stdout)` and `stderrBytes, _ := io.ReadAll(stderr)` discard errors. If `ReadAll` fails partway (e.g., device disconnect), the handler proceeds with partial data and may misinterpret the result. The same pattern appears in `handlers_apps.go:262-263` and `handlers_apps.go:276`.

**Fix:** At minimum, log the error:
```go
stdoutBytes, err := io.ReadAll(stdout)
if err != nil {
    slog.Warn("apk: stdout read incomplete", "device", serial, "error", err)
}
```

### WR-04: DownloadFolder uses request context for long-running recursive ls + pull

**File:** `internal/api/handlers_files_folder.go:233`
**Issue:** `DownloadFolderForTest` uses `r.Context()` for the `ShellRunRaw` recursive ls call and subsequent `SyncPullWriter` calls (line 303). If the client disconnects, the context is cancelled mid-transfer. Unlike upload-folder (which uses `context.Background()` with a timeout), download-folder has no bounded context. For a large directory, this could leave the device-side process in an indeterminate state. The pattern used in handlers_apk.go and handlers_apps_backup.go is to use `context.Background()` with a timeout for long-running ADB operations.

**Fix:** Add a bounded context similar to the upload-folder pattern:
```go
timeout := 10 * time.Minute // or make configurable
dlCtx, cancel := context.WithTimeout(context.Background(), timeout)
defer cancel()
// Use dlCtx instead of r.Context() for all ADB calls
```

### WR-05: shellQuote does not handle paths containing single quotes robustly for all ADB implementations

**File:** `internal/api/handlers_files.go:225-227`
**Issue:** `shellQuote` replaces `'` with `'\''`. This is a standard POSIX shell quoting technique and is correct for most cases. However, the function is called on paths that have already been validated and cleaned by `ValidateDevicePath`. Since `ValidateDevicePath` only allows alphanumeric characters and path separators within allowed base paths, the risk is minimal. This is a defense-in-depth note rather than a vulnerability.

**Fix:** No change needed; this is documented as belt-and-braces per the handler comments.

### WR-06: handlers_files_browse.go RenameFile uses request context for ShellV2Stream

**File:** `internal/api/handlers_files_browse.go:214`
**Issue:** `RenameFileForTest` passes `r.Context()` to `ShellV2Stream`. The `mv` command is typically fast, so this is low risk. However, the 5-second exit code timeout on line 228 means if the client disconnects, the context cancellation could leave the ShellV2Stream goroutine in an indeterminate state. This is inconsistent with the pattern used in other write handlers (apk install, backup, uninstall) which use `context.Background()` with a timeout.

**Fix:** Consider using `context.Background()` with a short timeout for consistency with other write handlers:
```go
renameCtx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
defer cancel()
stdout, stderr, exitCh, err := runner.ShellV2Stream(renameCtx, serial, cmd)
```

### IN-04: UploadFolderForTest writes HTTP 200 before validating tar body

**File:** `internal/api/handlers_files_folder.go:92-95`
**Issue:** The handler writes `w.WriteHeader(http.StatusOK)` and sets NDJSON content type before processing any tar entries. This means the client always receives a 200 status code, even if the tar body is malformed or the path is invalid. **Fix:** No change needed; this is intentional per the documented contract.
