## Decisions Made
- Original `hdr.Name` checked for `..` and leading `/` BEFORE `path.Clean` — `path.Clean("/" + hdr.Name)` resolves `..` away and strips absolute prefix, so checking only the cleaned path would miss traversal entries that the test suite expects to be rejected
- `parseLSRecursive` placed in `handlers_files_folder.go` rather than `ls_parse.go` since it's a download-folder-specific helper not reused elsewhere
- DownloadFolder does NOT acquire WriteInFlight (read op per Pitfall 9; only UploadFolder gets the single-flight gate)
- X-Skipped-Entries header uses two-pass approach: pass 1 scans for non-regular entries to populate the header, pass 2 emits tar entries; headers committed before `tar.NewWriter` call to ensure X-Skipped-Entries lands in the response

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Enhanced tar traversal rejection to check original hdr.Name**
- **Found during:** Task 1 test execution
- **Issue:** The plan's `path.Clean("/" + hdr.Name)` + `strings.Contains(cleanRel, "..")` check does not catch `../../../etc/passwd` because `path.Clean` resolves `..` segments away (turns it into `/etc/passwd`). Similarly, `/absolute/etc/passwd` entries lose the leading `/` after `path.Clean`. The test suite (TestUploadFolder_TarTraversalEntries) expects these to produce UNSUPPORTED_ENTRY lines with zero SyncPushReader calls, but the plan's code would accept them as safely sandboxed under root.
- **Fix:** Added two pre-Clean checks: (1) `strings.Contains(hdr.Name, "..")` rejects any entry with `..` in the original name, and (2) `strings.HasPrefix(hdr.Name, "/")` rejects absolute path entries. Both emit UNSUPPORTED_ENTRY NDJSON lines and skip the entry. The post-Clean `..` check is retained as belt-and-braces.
- **Files modified:** handlers_files_folder.go
- **Verification:** TestUploadFolder_TarTraversalEntries passes with pushCount == 1
- **Committed in:** 4627247
