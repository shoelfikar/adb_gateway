<objective>
Fix two compounding bugs that cause WebSocket code 1006 on /video and /audio endpoints: (1) subscribeAndRelay never calls ws.CloseRead(), preventing pong processing, ping timeouts, ReadLimit enforcement, and close-frame handling; (2) http.Server WriteTimeout (65s) persists on the hijacked net.Conn after WebSocket upgrade, causing writes to fail after 65s.

Purpose: Gaps 1 and 2 share the same root cause — write-only WS handlers (/video, /audio, /logcat) never process inbound control frames. coder/websocket requires concurrent Read() calls; CloseRead() is the correct API for write-only handlers. The HTTP server timeout is a secondary but critical bug that causes hard connection drops after 65s.

Output: Stable long-lived WebSocket connections for /video, /audio, and /logcat; corrected test coverage for ping/pong and ReadLimit.
</objective>

    **Bug 1 — Missing ws.CloseRead() in subscribeAndRelay (internal/api/ws_helpers.go):**

    In `subscribeAndRelay()`, immediately after the `defer cancel()` line and before the relay loop starts, add:

    ```go
    // Read-side goroutine: processes ping/pong and close frames.
    // Required by coder/websocket — without this, pongs are never dispatched
    // and Ping() blocks forever, causing idle-timeout disconnections (code 1006).
    ws.CloseRead(ctx)
    ```

    `CloseRead(ctx)` starts a background goroutine that reads and discards data messages while responding to ping/pong and close frames. The context cancellation from pingLoop (idle timeout) or parent cancellation will stop it.

    **Bug 1b — Missing ws.CloseRead() in StreamLogcat (internal/api/handlers_logcat.go):**

    In `StreamLogcat()`, after `applyWSDefaults(ws, cfg)` and before the snapshot replay loop (the `for _, line := range snapshot` loop), add:

    ```go
    // Read-side goroutine: processes ping/pong and close frames.
    // Required because StreamLogcat's main loop writes but never reads.
    ws.CloseRead(ctx)
    ```

    Keep the existing `pingLoop` goroutine — `CloseRead` and `pingLoop` serve different purposes:
    - `ws.CloseRead(ctx)` handles reading control frames (pong responses, close frames) — this is the missing piece that was causing the bug
    - `pingLoop(ctx, ws, cfg)` continues to send outbound pings and detect idle timeouts
    - Both run concurrently; when pingLoop returns (idle timeout), it cancels ctx, which closes both goroutines

    StreamLogcat's main loop reads from the `ch` channel (logcat subscriber), NOT from the WebSocket. So StreamLogcat IS a write-only handler from the WS perspective — same bug as subscribeAndRelay. The fix is identical: add `ws.CloseRead(ctx)`.

    **Bug 2 — HTTP server WriteTimeout/ReadTimeout (cmd/gateway/main.go):**

    In `cmd/gateway/main.go`, change the `http.Server` configuration from:

    ```go
    ReadTimeout:  15 * time.Second,
    WriteTimeout: 65 * time.Second,
    IdleTimeout:  60 * time.Second,
    ```

    To:

    ```go
    ReadHeaderTimeout: 15 * time.Second,
    // WriteTimeout is intentionally 0 (no deadline). After WebSocket Hijack(),
    // the net.Conn retains any WriteTimeout deadline set by http.Server,
    // causing WS writes to fail after 65s. Setting WriteTimeout: 0 prevents
    // this. WS idle/ping timeouts are handled by our own pingLoop, not HTTP
    // timeouts. ReadHeaderTimeout is safe because it only covers reading
    // request headers, not the body (which for WS is the upgraded connection).
    WriteTimeout: 0,
    IdleTimeout:  60 * time.Second,
    ```

    `ReadHeaderTimeout` (Go 1.8+) only sets a deadline on reading request headers, not the body. After the WebSocket upgrade, the body is the WS stream which should have no HTTP-level deadline. This is the standard pattern for Go servers that handle long-lived WebSocket connections.

    Note: `WriteTimeout: 0` means no write deadline. For REST endpoints (non-WS), response write time is unbounded at the HTTP level. This is acceptable because (a) the session creation endpoint already has its own context timeouts (ADB push/reverse/accept), and (b) regular API responses are fast. The real risk of slow writes is only on WS connections, which are handled by our own pingLoop idle detection.
